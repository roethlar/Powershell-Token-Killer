## Architecture

New solution `siem/PtkSiem.slnx` (house `.slnx` format), all projects `net10.0`, `Nullable` + `ImplicitUsings` enable, no analyzers/AOT (matches `server/` conventions):

- **`siem/PtkSiemReceiver/PtkSiemReceiver.csproj`** (`Microsoft.NET.Sdk.Web`, Exe). Generic Host + `BackgroundService` composition mirroring `server/PtkMcpServer/Program.cs`. Two Kestrel endpoints:
  - **Ingest endpoint** (mTLS required): `POST /v1/logs`, OTLP/HTTP protobuf only (`Content-Type: application/x-protobuf`). Frozen client-cert policy per ingest listener: Kestrel `ClientCertificateMode.RequireCertificate`; chain validation via `X509ChainPolicy` with custom root trust pinned to the configured CA **bundle** (multiple CAs allowed so old+new overlap during rotation); clientAuth EKU + validity window enforced; revocation-check mode explicit in config, no silent fallback. S2 negative tests on all three OSes: missing cert, wrong CA, expired, wrong EKU, revocation-policy mismatch, old/new rotation pairs.
  - **Operator endpoint** (separate port, loopback-bound by default): query API + dashboard. A non-loopback operator bind is a **startup failure** unless an operator HTTPS certificate is configured — the bearer token never travels plaintext off-host (startup validation + network integration test, S1/S5).
- **Wire layer**: compiles the canonical proto via `<Protobuf Include="../../server/PtkMcpServer/Protos/audit_otlp.proto" Link="Protos/audit_otlp.proto" GrpcServices="None" />` with `Google.Protobuf` 3.35.1 + `Grpc.Tools` 2.82.0 (`PrivateAssets=all`) — same packages/pattern as the producer, independently compiled, no `OpenTelemetry.*` NuGet. PTK's copy of the proto is never edited by this plan.
- **Ingest pipeline** (per request, in order):
  1. Parse `ExportLogsServiceRequest`; malformed/oversized ⇒ HTTP 400, never an ack. Request size cap configurable (default 1 MiB).
  2. Extract the single log record (producer sends one record per request); validate schema version (v1/v2/v3 accepted), retained attribute names/types, and the four `ptk.host.*` rules.
  3. **Strict record validation** (independent re-specification, no PTK runtime code imported): recompute `event_hash` as SHA-256 over the exact reconstructed pre-hash bytes (the `AuditSpoolRecordCodec` envelope semantics), verify exact envelope fields, canonical UUID forms, sequence rules keyed by `producer.supervisor_boot_id`, expected next sequence + `prev_hash` against the stored chain head, and equality between body values and every projected OTLP attribute — one mutation test per check (S4). Two carve-outs are evaluated **before** chain-position rejection: (a) a byte-identical duplicate of an already-committed row (event ID + `event_hash` match) short-circuits to check 4's idempotent success ack before any next-sequence/head comparison — a legitimate at-least-once replay is by definition no longer "next" and must never be chain-rejected (S4 test: commit N, let the head advance, replay N byte-identically ⇒ idempotent `200` ack, not `400`); (b) while a gap for the record's `producer.supervisor_boot_id` is `open`, an otherwise-valid record whose sequence lies beyond the gap is not `400`-rejected — it is committed flagged `post-gap` with `prev_hash` continuity enforced against the previous stored post-gap record for that boot (the first post-gap record anchors the sub-chain at the gap boundary) and acknowledged with the frozen `200` success ack, pending operator disposition — this is the storage path the S6 gap-resume protocol depends on. Every other failure ⇒ permanent rejection (HTTP `400`, see response table), never an ack. Before the `400` is written, a rejected-attempt/quarantine row is committed durably in its own transaction (attempt evidence, claimed hashes, implied gap) so chain-break/gap alerting has durable input without acknowledging the invalid record.
  4. Duplicate detection by event ID + hash — evaluated **before** chain-position/next-sequence rejection (the numbering is presentation order, not evaluation order): identical replay ⇒ idempotent success ack without a second row (at-least-once delivery, discovery matrix row 3), even after the chain head has advanced past it; same event ID with different bytes ⇒ rejection.
  5. **Durable commit**: one serialized SQLite transaction (`BEGIN IMMEDIATE`) that re-reads and re-validates the stored chain head *inside* the transaction (step 3's earlier read is advisory only), inserts the raw request bytes + extracted columns, and conditionally advances the exact prior head — chain-head validation and insertion are never split across transactions; a unique `(boot_id, sequence)` constraint backstops fork prevention. `PRAGMA journal_mode=WAL; PRAGMA synchronous=FULL;` so the commit is fsync-durable before the response is written. Fork test (S3): two simultaneous candidates for the same next sequence — exactly one event commits, the loser is rejected/quarantined.
  6. **Ack**: only after the transaction commits, write the frozen success response: HTTP `200`, parameter-free `Content-Type: application/x-protobuf`, body = serialized empty `ExportLogsServiceResponse` with `partial_success` unset (the valid nonrejecting ack of `server/AUDIT-EXPORT.md:87-97`, recognizable by PTK's exporter classifier, `server/PtkMcpServer/Audit/AuditOtlpHttpExporter.cs:248`).
  - Disk-full / `SQLITE_FULL` / any commit failure or backpressure ⇒ HTTP `503` + protobuf `google.rpc.Status` + `Retry-After` — **never a false 200** (discovery `:132-134`). 503 is the transient signal PTK's existing retry schedule honors.
  - **Frozen response table** (every row tested against PTK's real exporter classification, S2): `200` = the exact success ack in step 6, nothing else; `400` + protobuf `google.rpc.Status` = permanent single-record validation failure (malformed, schema violation, strict-validator failure, mismatched duplicate); `503` + protobuf `google.rpc.Status` (+ `Retry-After`) = transient storage/backpressure failure. Because each PTK request carries exactly one record, whole-request rejection uses HTTP error codes per the [OTLP spec](https://opentelemetry.io/docs/specs/otlp/); `partial_success` is never used in this contract.
- **Store** (`siem/PtkSiemReceiver`, `Storage/`): SQLite file under a configurable data root (through S3, the configured SQLite file's immediate parent is the data root). Tables: `events` (unique event ID, a unique `(boot_id, sequence)` composite constraint, schema version, raw protobuf bytes, event hash, prev hash, received UTC, extracted query columns + indexes), `chains` (per-boot chain head state), `gaps`/`alerts`, `meta` (receiver identity, schema migration version). Raw bytes are the evidence of record; extracted columns are derived and rebuildable. **Receiver custody ledger** (the owner-required receiver-side hash chain, distinct from PTK's producer chain): a versioned `custody` table — receipt sequence, prior receipt hash, SHA-256 over a deterministic framing of the exact raw request bytes + immutable receiver metadata (receipt UTC, client-cert thumbprint, remote endpoint, disposition) — written atomically in the same transaction as the event insert; retention tombstones, quarantine rows, and alert lifecycle records (creation, acknowledgement/state change, deletion — each carrying rule/config identity, written in the alert-persistence transaction) chain through it. PTK's `event_hash`/`prev_hash` cover only the producer's source JSON body; the custody ledger is what makes receiver-side tampering (rows, alerts, tombstones) evident. Mutation tests cover raw bytes, metadata, tombstones, alert deletion/modification, and quarantine mutation (S4). **Independent custody witness** (a hash chain alone cannot detect whole-store rewrite or tail truncation — `server/AUDIT-EXPORT.md:251-254`): the receiver appends custody-head checkpoints (receipt sequence + head hash + UTC) at every clean shutdown and on a configurable cadence to an append-only checkpoint location **outside** the SQLite data root with stricter, separately administered ownership (startup-validated like the other protected paths), The local checkpoint stream is receiver-written, so its guarantee is explicitly bounded: it detects store tampering by any actor lacking write access to the checkpoint location, and whole-store rewrite or tail truncation by anyone who cannot also forge checkpoints — it cannot, by itself, defeat an adversary holding the receiver identity who rewrites store and subsequent checkpoints together, and an unwitnessed suffix (records newer than the last checkpoint) is protected only up to the checkpoint cadence. That residual tier requires off-box anchoring: when configured (webhook or write-once file drop), the receiver ships every checkpoint head off-box; S7 install docs make configuring an anchor the recommended default and specify the manual alternative (operator records the published head out-of-band on a fixed cadence), and the S7 threat-model table states plainly that coordinated store+checkpoint rewrite and unwitnessed-suffix loss by a receiver-identity adversary are detected **only** with an off-box anchor or manual attestation in place. Startup and periodic verification recompute the ledger against the newest retained checkpoint (and the newest anchored head when off-box anchoring is configured) and alarm on any mismatch or head regression. S4 must include full-database rewrite, tail-truncation, and checkpoint-regression tests that alarm. No checkpoint or file-drop-anchor path exists through S3; the later slice that introduces either path must route it through S3H's protected-path boundary before its first read or write and extend S3H's negative matrix. S3H does not add witness configuration or behavior.
- **Query API + dashboard** (`Web/`): read-only minimal-API endpoints (events by time/type/session/boot filters, event detail with chain context, chain status, gap and alert lists) rendered by embedded static htmx pages, plus the **alert-lifecycle API** — the sole writer of alert custody events: authenticated endpoints to acknowledge/annotate/close an alert with explicit transition rules (`open → acknowledged → closed`; no other transitions; alert rows are never deleted by this API — closure is a state change, and the only alert-row deletion is retention-driven and itself custody-recorded), each transition committed in the same transaction as its custody-ledger entry recording who/when/prior-state. S6 tests drive every legal transition and reject every illegal one; S4 mutation tests tamper with a persisted transition record and must detect it. Operator auth: bearer token from the config file (0600-perm), endpoint loopback-bound unless explicitly configured otherwise — read authorization distinct from the ingest path (matrix rows 7/8).
- **Alert rules** (`Alerting/`): declarative rules in the config file — event-type/attribute match, chain-break, gap-detected, ingest-rate threshold. Chain-break/gap alerts are fed by the durable quarantine/gap rows written at rejection time (see ingest pipeline step 3), so a rejected record still produces a durable alert. A receiver-local, separately administered gap-resolution state machine governs resumption after PTK's operator gap disposition (`server/AUDIT-EXPORT.md:176`): states `open → dispositioned(resolved | accepted-loss) → resumed`, persisted on the `gaps` row and chained through the custody ledger; the sole transition authority is an authenticated operator API call (who/when/disposition recorded); resume boundary — while `open`, later valid records for that boot are stored but flagged `post-gap` (never silently re-anchored), and on disposition the chain resumes from the first stored post-gap record; resolution never acknowledges the invalid record and preserves the missing-hash evidence. S6 test: reject → operator disposition → restart → resume, proving the state survives restart. Actions: dashboard alert list + optional webhook POST with retry. Evaluation is decoupled from the ack path but **crash-safe**: the same transaction that commits a quarantine/gap/event row durably enqueues an alert-evaluation work item stamped with the SHA-256 of the startup-frozen rule configuration active at enqueue time; a persisted evaluation cursor plus idempotent startup replay guarantees every committed work item eventually yields its alert exactly once, and replayed items evaluated after a restart record both config hashes (enqueue-time and evaluation-time) on the resulting alert so a rule change across a crash is evident, never silent — the S6 kill test asserts the recorded config identity on the post-restart alert. S6 kill test: crash after ingest/quarantine commit but before alert persistence — the alert exists after restart.
- **Retention** (`Storage/RetentionService`): age/size-based purge as a `BackgroundService`; retention settings in config (matrix row 8). **Frozen tombstone commitment** (schema versioned with the store): each purge writes — atomically in the same transaction as its deletes, with rollback on partial failure — a tombstone row committing to the purged per-boot sequence range, boundary hashes (last retained `prev_hash`/`event_hash` on each side), purged-record count, a deleted-record commitment (rolling SHA-256 over the purged records' `event_hash` values in sequence order), and custody predecessor/successor linkage, so producer-chain and custody verification succeed over retained history without the raw bytes. Tests (S4): purge → restart → full chain + custody verification passes; tombstone boundary mutation alarms; partial-purge rollback leaves the store verifiable.
- **Config**: env var `PTK_SIEM_CONFIG` → one validated JSON file, frozen at startup, modeled on `AuditExportConfigurationLoader`/`AuditStartupConfiguration` (`server/PtkMcpServer/Audit/AuditExportConfiguration.cs`, `AuditStartupConfiguration.cs`): ingest bind address, server certificate + key, client CA, operator bind address + token, SQLite path, retention, alert rules. Strict validation, actionable failures, no fallback defaults for security-relevant fields.
- **Separate principal + off-host custody** (matrix rows 1/7; `server/AUDIT-EXPORT.md:205`; `.agents/decisions.md:304`): an **anchored** deployment requires a distinct receiver host (or equivalently isolated machine) under a separately administered principal — records must leave the PTK source machine. Same-host installs, including under a second OS account, are documented as development/non-anchoring (`PtkAuditAdmin`-style same-user layouts likewise). The receiver runs as its own OS account with data root and config owned by that account. **Startup enforcement, fail closed** (the `server/AUDIT-EXPORT.md:48` precedent): config, database, `-wal`/`-shm` sidecars, data directory, the custody checkpoint location once introduced, **every referenced TLS material file** (ingest server certificate + private key, client CA bundle(s), operator HTTPS certificate + key when configured), and the immediate parents of all of these must be owner-only. On POSIX, the effective process user owns each protected path, directories are exactly `0700`, and files are exactly `0600`; macOS extended ACLs that grant another principal are refused. On Windows, the current receiver SID owns each path and its protected DACL contains exactly one non-callback, non-inherited allow ACE granting that SID `FullControl`. Existing config, TLS, database, sidecar, and directory objects are verified without permission repair; receiver-created database and sidecar objects are created/protected and then verified. Every existing lexical path component is checked without following it and any symlink, junction, mount-point reparse point, or other reparse redirect is refused. Enforcement completes before either listener starts, with negative tests on every OS (including permissive mode/DACL, link/reparse, and replaced/swapped cases for each config/PEM role), not documentation-only. Install docs (S7) cover Windows service account, and dedicated users on macOS/Linux.
- **`siem/PtkSiemReceiver.Tests/PtkSiemReceiver.Tests.csproj`**: xUnit 2.9.3 / runner 3.1.4 / Test SDK 17.14.1 / coverlet 6.0.4, `<Using Include="Xunit" />`, flat `*Tests.cs` files, nested `private sealed class XxxFixture : IDisposable`, `[CollectionDefinition(..., DisableParallelization = true)]` for process/port tests — the `server/PtkMcpServer.Tests` house style.

### S3H implementation contract

1. **Scope and protected manifest.** Add a SIEM-local protected-path boundary; do not share or import PTK runtime code. It covers the absolute `PTK_SIEM_CONFIG` file, ingest certificate and private key, every client-CA bundle, optional operator HTTPS certificate and key whenever configured, the absolute SQLite database path, derived `<database>-wal` and `<database>-shm` names, and the deduplicated immediate parents/data directory. Reject relative configured paths. The custody checkpoint and optional file-drop anchor are absent through S3 and remain out of scope; their owning later slice must add them to this manifest before first use.
2. **Protected config read comes first.** Before parsing the operator token or any referenced path, open the config through a platform no-follow boundary, verify that the retained object is the expected regular file with a protected immediate parent, enforce the frozen size bound while reading from that retained object, and verify its stable file identity before and after the read. On Linux/macOS, use non-following open plus `lstat`/`fstat` device-and-inode comparison; on Windows, retain a `CreateFile` handle opened with `FILE_FLAG_OPEN_REPARSE_POINT` and compare reparse attributes, file identity, and final path. A replacement must either leave the originally verified bytes in use or fail startup; it must never cause replacement bytes to be accepted. Existing config permissions/DACLs are never changed.
3. **TLS inputs are identity-stable bytes, not reopened paths.** After config parsing and before certificate construction, apply the same non-mutating protected read to every configured PEM role and parent. The active ingest certificate/key and CA loader consumes those verified bytes (`CreateFromPem`/collection byte or character APIs) rather than reopening a path after validation, and clears private-key/config byte buffers when ownership ends. Optional operator HTTPS paths are protected and securely read even though S5 has not yet added that listener; S3H does not pull that listener or its remaining certificate-suitability behavior forward. Each input has an explicit size cap, and wrong-kind or empty-input failures remain sanitized startup errors.
4. **SQLite is protected around creation and WAL startup.** Before SQLite opens, refuse linked/reparse components and insecure existing database/`-wal`/`-shm` objects, and verify the existing data directory without mutating it. Snapshot which artifacts were absent, atomically precreate each absent database/sidecar name owner-only inside that already protected directory, and give those exact identities to SQLite; this is required on Windows because a later pathname DACL retrofit would leave a creation-time exposure. After WAL policy and migrations materialize the live artifacts, reverify all database/sidecar names, types, identities, ownership, and protection before startup can succeed. Existing insecure objects fail without repair or content mutation. Keep the writer open after this boundary so SQLite cannot recreate an unchecked live sidecar before shutdown. The protected parent plus before/after identity checks defend against other principals; do not claim resistance to a malicious actor already running as the receiver identity.
5. **Exact platform policy.** POSIX verification uses numeric effective-UID ownership and exact `0700` directory/`0600` file modes; on macOS it also rejects an extended ACL. Secure creation uses an atomic owner-only create mode, followed by verification. Windows verification is non-mutating for existing objects and requires the current token SID as owner plus a protected DACL containing exactly one non-callback allow ACE for that SID with `FullControl`; receiver-created objects receive that descriptor and are read back for exact verification. Link checks walk every existing lexical component without resolving it and cover dangling links, file symlinks, directory symlinks, junctions, mount points, and other reparse points.
6. **Startup ordering and failures.** Production startup cannot bind either endpoint until config, all configured TLS inputs, and the SQLite pre/post-open boundary have completed. All violations produce stable, sanitized `SiemReceiverConfigurationException` or `SiemReceiverStartupException` codes without emitting the operator token, PEM material, or other secret content. No hardening failure is downgraded to a warning or repaired for an existing operator-provisioned object.
