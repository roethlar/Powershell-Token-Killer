# sd1-2: paired-backtick scan can be closed by unrelated later text

**Severity**: MEDIUM — deterministic false positive on valid PowerShell (violates the frozen precision-first rule), though the input class is narrow.
**Status**: In progress
**Branch**: master (direct)
**Commit**: (filled in after commit)

## Evidence
`src/PwshTokenCompressor.psm1:677` (slice-1 head `6694dc5`): after finding an element that opens with `` `letter ``, the scan regexes over `$Script.Substring(offset)` to end of script, so any later backtick — a second legitimate escape, a comment — can close the pair.

## Predicted observable failure
`Write-Output `tColumn` Name` (tab escape + escaped space, one legitimate argument) is reported as bash command substitution.

## What
The pair rule is not token-bounded; the raw-text tail scan crosses element boundaries.

## Approach
Replace the tail scan with element-extent-bounded pairing: self-closed = extent exactly `` `word...` ``; otherwise an opener extent (starts `` `letter ``, no further backtick) must be closed by a LATER element extent that ends with its only backtick. Interior-backtick extents like `` `tColumn` Name `` match neither shape.

## Files changed
- `src/PwshTokenCompressor.psm1` — extent-bounded backtick pairing
- `tests/PwshTokenCompressor.Tests.ps1` — `` `tColumn` Name `` false-positive test

## Guard proof
`tests/PwshTokenCompressor.Tests.ps1::'ignores escape sequences joined by an escaped space'` — FAILS with the tail scan restored, PASSES with extent bounding. Existing IN tests (`` echo `date` ``, `` echo `date +%s` ``) prove no recall loss.

## Coder dispute (if any)
None.

## Known gaps
An opener followed by a degenerate trailing-backtick-at-EOF element could still pair; that shape is outside the frozen legitimate-backtick set (lone escape, line continuation).
