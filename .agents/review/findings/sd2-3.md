# sd2-3: Background dialect refusals do not refresh the idle deadline

**Severity**: LOW — an actively used server can idle-exit immediately after
serving a refusal; only near the PTK_IDLE_EXIT_SECONDS boundary.
**Status**: Open
**Branch**: master (direct, per recorded precedent)
**Commit**: (filled in after commit)

## Evidence
`server/PtkMcpServer/Tools/InvokeTool.cs` background branch: a detected
script returns via `TryGetBackgroundDialectRefusalAsync` BEFORE
`TryGetCurrentLocationAsync` (the first call that routes through
InvokeAsync and touches `LastActivityUtc`).
`server/PtkMcpServer/RunspaceHost.cs`: the new method never updates
`LastActivityUtc`; `server/PtkMcpServer/IdleWatchdog.cs` reads it.

## Predicted observable failure
A `background=true` call refused just before the idle deadline does not
count as activity; the watchdog can stop a server the user is actively
talking to, right after it answered.

## What
A refusal is a served user call — every other user-call boundary
(InvokeAsync, ResetAsync) stamps `LastActivityUtc`; this new one does not.

## Approach
Stamp `LastActivityUtc` at the top of `TryGetBackgroundDialectRefusalAsync`
— the user-call boundary — matching InvokeAsync/ResetAsync. Guard: the
method's invocation advances `LastActivityUtc`.
