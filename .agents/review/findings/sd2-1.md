# sd2-1: Refusal teaches POSIX apostrophe escaping to a PowerShell parser

**Severity**: MEDIUM — the D1-required apostrophe note sends the model into a
parse error on the plan's own recovery path.
**Status**: In progress
**Branch**: master (direct, per recorded precedent)
**Commit**: (see final line of this doc's Files changed once committed)

## Evidence
`server/PtkMcpServer/RunspaceHost.cs` (FormatDialectRefusal, bash-available
leg) advertises "a literal ' inside the wrap must be written as '\''" — the
POSIX idiom for a *bash* outer layer. But the wrap `bash -lc '...'` is
submitted to ptk_invoke as a **PowerShell** command line, where the
single-quoted wrap is a PowerShell string literal.
`server/PtkMcpServer.Tests/ShellDialectWiringTests.cs`
(Refusal_recovery_guidance_is_platform_aware) blesses the wrong substring.

## Predicted observable failure
Following the advice for a script containing an apostrophe (e.g.
`bash -lc 'echo it'\''s'`) fails PowerShell parsing before bash starts.
Master-verified 2026-07-09: ParseInput reports 1 error,
`TerminatorExpectedAtEndOfString`. The correct composed escape —
double for PowerShell, backslash for bash (`bash -lc 'echo it\''s'`) —
parses clean and prints `it's` (verified live).

## What
The refusal's escaping note is correct for the wrong outer shell. Two
quoting layers compose here (PowerShell string literal, then bash script
text) and the note only covers a bash outer layer.

## Approach
Reword the bash-available recovery line to teach the composed escape with a
concrete worked example. Strengthen the guard: assert the refusal carries
the exact example, and execute that example verbatim (raw=true) asserting
it prints `it's` — a wrong example fails the battery on any bash box.

## Known gaps
The example covers an apostrophe outside bash-level quotes; apostrophes
inside a bash single-quoted region still need bash-side care. The note
teaches the common case; deeper bash quoting is the wrapped script's own
dialect, out of the refusal's scope.
