# sd1-1: clean-parse classifiers lack the shadowing guard the router already has

**Severity**: MEDIUM — once wired (slice 2), a session-defined `function export`/`local`/`source` is refused even though it resolves and runs correctly in the warm runspace.
**Status**: In progress
**Branch**: master (direct)
**Commit**: (filled in after commit)

## Evidence
`src/PwshTokenCompressor.psm1:643` (slice-1 head `6694dc5`): the `export`/`local`/`source`/`VAR=` classifiers key on name shape alone. `Resolve-PtcInvokeScript` (same file) already guards the equivalent decision with `$ExecutionContext.InvokeCommand.GetCommand(...)`.

## Predicted observable failure
`function export { param($Assignment) "ran:$Assignment" }; export X=1` executes successfully in the warm runspace, yet the detector returns `the bash 'export' builtin` — the refusal path would reject valid PowerShell.

## What
The CommandNotFound-class classifiers assert a failure that cannot happen when the name actually resolves in the session.

## Approach
Compute the candidate label first; before returning it, resolve the name with the same intrinsic the router uses. If it resolves to anything, the predicted failure is impossible — skip the finding. `set` is intentionally exempt: it resolves (alias) in every pwsh by design and is keyed to its argument shape instead (frozen slice-0 decision).

## Files changed
- `src/PwshTokenCompressor.psm1` — resolution guard around the CommandNotFound-class returns
- `tests/PwshTokenCompressor.Tests.ps1` — shadowed-`export` test (define `global:export`, expect null; remove, expect finding)

## Guard proof
`tests/PwshTokenCompressor.Tests.ps1::'stays silent when the bash name is shadowed by a real session command'` — FAILS with the guard reverted, PASSES with it restored.
