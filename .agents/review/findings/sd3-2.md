# sd3-2: Elided job polls advise a recovery control ptk_job does not have

**Severity**: MEDIUM — the in-band marker misdirects at the exact recovery
moment; following it can duplicate expensive or side-effecting work.
**Status**: In progress
**Branch**: master (direct, per recorded precedent)
**Commit**: (filled in after commit)

## Evidence
`src/PwshTokenCompressor.psm1` elision markers say "rerun with raw=true";
`server/PtkMcpServer/Tools/JobTool.cs` action=output always shapes via
`ShapeTextAsync` (→ Compress-PtcOutput → Limit-PtcPassthrough, 400
lines / 40KB) and exposes NO raw parameter; `JobManager.ReadOutput`
advances the poll offset past the raw chunk, so the elided middle never
comes back on later polls. Coder-verified by code reading: a >40KB poll
chunk hits the marker with no raw control in reach.

## Predicted observable failure
A model polling an oversized job output follows the marker: re-issuing
the job with `background=true, raw=true` starts a SECOND job (duplicate
expensive/side-effecting work) whose polls are still shaped and elided;
meanwhile normal next-offset polling has permanently skipped the middle.
The actual recovery — the complete raw log file, whose path status
reports — is not named at the moment of elision.

## What
The marker's recovery advice is composed for the ptk_invoke context and
is wrong in the ptk_job context; the honest recovery there is the job's
raw log on disk.

## Approach
JobTool's output action appends an in-band correction when the shaped
poll carries the elision marker: raw=true does not apply here, the
complete raw log is at <OutputPath>. The substring coupling to the marker
wording is pinned by the guard test, so a marker reword fails the battery
instead of silently dropping the note.

## Files changed
- `server/PtkMcpServer/Tools/JobTool.cs` — elided polls append the
  log-path recovery note.
- `server/PtkMcpServer.Tests/RawUsageTests.cs` — new
  Oversized_job_poll_names_the_log_as_the_recovery_not_raw.

## Guard proof
- `RawUsageTests::Oversized_job_poll_names_the_log_as_the_recovery_not_raw`
  — red leg: with the JobTool change stashed, the poll carries the marker
  but no correction → FAILS; restored → PASSES.

## Coder dispute (if any)
None — admitted; mechanics verified against JobTool/JobManager/module
code before triage.

## Known gaps
The module marker itself still names raw=true inside the appended-note
context; the correction is adjacent and in-band rather than replacing the
marker (replacing would require plumbing a per-context hint through
Compress-PtcOutput — not taken without need).
