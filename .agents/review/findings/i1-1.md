# i1-1: mixed-type header grows unbounded with unique type names

**Severity**: MEDIUM — default shaping can flood context on streams with
many distinct type names; no corruption, no throw.
**Status**: Verified
**Branch**: master (direct, per the recorded codex-loop precedent)
**Commit**: (filled in at commit)

## Evidence
`src/PwshTokenCompressor.psm1` (Compress-PtcGenericObject as of 66a53df):
the `mixed:` header joined ALL unique `PSObject.TypeNames[0]` values; the
table and samples are capped, the type list was not.

## Predicted observable failure
A heterogeneous non-string stream with many distinct type names
(deserialized/proxy/custom PSTypeName objects) emits a header line that
grows with the unique-type count — unbounded output from the compressor
whose contract is bounding.

## Approach
Header shows the first 3 type names plus `+N more`.

## Files changed
- `src/PwshTokenCompressor.psm1` — header construction.

## Guard proof
- `tests/PwshTokenCompressor.Tests.ps1::bounds the mixed-type header to a
  few names` — 5 distinct PSTypeNames; asserts the `+2 more` suffix and
  the absence of the 4th type name. FAILS with the unbounded join
  reinstated; PASSES with the cap.

## Coder dispute (if any)
None.

## Known gaps
Individual type names are not width-limited; three .NET type names are
bounded in practice. Writing this finding's guard also exposed a separate
defect (Select-Object TypeNames mutation) fixed in its own follow-up
commit.
