# sd3-4: Elision can lengthen text, so the shortening gate suppressed genuine notes

**Severity**: MEDIUM — restores the sd3-2 recovery failure for an ordinary
boundary case (genuinely elided poll, no honest recovery named).
**Status**: In progress
**Branch**: master (direct, per recorded precedent)
**Commit**: (filled in after commit)

## Evidence
`server/PtkMcpServer/Tools/JobTool.cs` (round-1 sd3-3 fix) required
`shaped.TrimEnd().Length < text.TrimEnd().Length`, but
`Limit-PtcPassthrough` replaces near-boundary content with a marker LONGER
than what it removed. Reviewer probe, coder-reproduced verbatim
2026-07-10: 401 one-character lines → in 801 chars, out 872 chars, with a
genuine elision marker present.

## Predicted observable failure
A genuinely elided job poll omits the corrective raw-log note, leaving
only the inapplicable raw=true advice after the offset has advanced past
the missing content; following it can duplicate side-effecting work —
exactly the sd3-2 failure the note exists to prevent.

## What
"Elision shortens text" is false at the boundary; any length-based
inference of elision is unsound (sd3-3's ANSI tail broke it in the other
direction in the same pass).

## Approach
Fixed jointly with sd3-3's completion in one redesign (one coherent
change; the two findings are the two failure directions of the same
heuristic): the elision hint rides INTO shaping (`-ElisionHint` through
Compress-PtcOutput/Limit-PtcPassthrough, forwarded by ShapeTextAsync), so
the marker itself names the job's raw log exactly when the module elides.
No inference remains to be wrong. Detail: `.agents/review/findings/sd3-3.md`.
