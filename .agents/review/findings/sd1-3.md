# sd1-3: parse-fatal keys take bash evidence from comments and strings

**Severity**: LOW — the script is already unparsable either way; the harm is dialect guidance wrongly naming bash.
**Status**: Resolved (round 2 fix landed) — pending re-grade round 2
**Branch**: master (direct)
**Commit**: `ceae832` (round 1, kind-list blanking) + `4e6a223` (round 2, quoted fragments folded into Generic-kind string tokens blanked span-by-span; guard: 'takes no bash evidence from quoted fragments folded into Generic tokens (sd1-3 round 2)')

## Evidence
`src/PwshTokenCompressor.psm1:605` (slice-1 head `6694dc5`): the six parse-fatal checks pair the parser's error id with a regex over the whole raw `$Script`, so shape text inside a comment or string literal counts as evidence.

## Predicted observable failure
`Write-Output > # cat <<EOF` fails with an unrelated pwsh `MissingFileSpecification` (missing redirect target), but the comment text makes the detector return the heredoc finding — wrong guidance.

## What
Raw-text shape matching is not comment/string-aware.

## Approach
Blank the extents of Comment/StringLiteral/StringExpandable/HereString* tokens (offsets preserved) and run the six shape regexes over the blanked text. The heredoc key relaxes to `<<` since a quoted terminator (`<<'EOF'`) lexes its terminator as a string literal — bare `<<` outside strings/comments is itself never valid pwsh, and the error-id pairing still gates it.

## Files changed
- `src/PwshTokenCompressor.psm1` — token-blanked scan text for the parse-fatal branch
- `tests/PwshTokenCompressor.Tests.ps1` — comment-evidence false-positive test

## Guard proof
`tests/PwshTokenCompressor.Tests.ps1::'takes no bash evidence from comments'` — FAILS with raw-script matching, PASSES with blanked scan text. Existing heredoc IN tests (unquoted + quoted terminator) prove no recall loss.

## Coder dispute (if any)
None.

## Known gaps
Token streams beside parse errors can be partial; blanking only ever removes evidence, so degradation is toward a miss (today's behavior), never an over-match.
